# Portless Research Report

## Overview

Portless is an open-source CLI tool from Vercel Labs that replaces localhost port numbers with stable, named `.localhost` URLs. Instead of remembering that your API is on `localhost:3001` and your frontend is on `localhost:3000`, you run `portless myapp next dev` and access `http://myapp.localhost:1355`. It works by running a lightweight reverse proxy daemon on port 1355 that routes subdomain-based requests to the correct backend port, which it assigns randomly from the 4000-4999 range via the standard `PORT` environment variable.

Created on February 15, 2026, by Chris Tate (a Vercel employee), the project has accumulated 1,228 GitHub stars in just 4 days and shipped 7 npm versions (v0.0.0 through v0.4.0). The rapid velocity is notable: v0.4.0 added HTTP/2 + HTTPS support with automatic certificate generation and trust store integration -- a feature that normally takes weeks to implement correctly. Two PRs carry the "claude-code-assisted" label, confirming AI-assisted development contributed to the velocity. The tool is TypeScript-only, Apache-2.0 licensed, has a single runtime dependency (chalk), and includes a comprehensive test suite via Vitest.

What makes portless distinctive in 2026 is its explicit "agent-first" design philosophy. The tagline reads "For humans and agents," and the repo ships with both an AGENTS.md (for AI coding assistants working on portless itself) and a SKILL.md (an AI agent skill for using portless in projects). This directly addresses a real pain point: AI coding agents in monorepos frequently guess or hardcode wrong ports, causing test failures and wasted tokens. By giving each service a stable, named URL, agents can reliably reference `http://api.myapp.localhost:1355` instead of hoping `localhost:3001` is correct.

## Technical Analysis

**Architecture:** Portless uses a daemon-based reverse proxy model. `portless proxy start` launches a background process on port 1355 (or custom port). When you run `portless <name> <cmd>`, it picks a random port from the 4000-4999 range, sets `PORT=<random>` in the child process environment, registers the name-to-port mapping with the proxy via a state directory, and spawns your command. The proxy reads `.localhost` subdomain requests and routes them to the registered port.

**Source Code:** The codebase is small and focused -- 7 source files (cli.ts, proxy.ts, routes.ts, certs.ts, cli-utils.ts, utils.ts, types.ts, index.ts) totaling roughly 130KB including tests. The proxy implementation uses Node.js native HTTP/2 and HTTPS modules without heavy dependencies.

**HTTP/2 + HTTPS (v0.4.0):** The `--https` flag generates a local CA and server certificates, then prompts for sudo once to add the CA to the system trust store. Subsequent runs require no prompts. This solves the HTTP/1.1 six-connection-per-host bottleneck that affects dev servers serving many unbundled modules (Vite, Nuxt, etc.). Custom certs (e.g., from mkcert) are also supported via `--cert` and `--key` flags.

**State Management:** Routes, PID files, and port files are stored in `~/.portless` (ports >= 1024) or `/tmp/portless` (ports < 1024, for sudo scenarios). This is simple and appropriate for a local dev tool but means state doesn't survive reboots for the daemon.

**CI/CD:** GitHub Actions workflows exist. The test suite covers certs, CLI utilities, CLI commands, proxy behavior, and route management. The build uses tsup for bundling TypeScript.

**Platform Support:** macOS and Linux only. Windows support is actively requested (issue #15) and a community PR (#6) is open but not yet merged.

**Known Issues:**
- HTTPS fails with "md too weak" error on some configurations (#17)
- Running proxy on port 80 requires all apps to use sudo too (#16)
- Vite compatibility not yet documented (#12)
- No config file for default app names (#8, PR #13 in progress for `.portlessrc.json`)
- No way to specify a desired application port (#11)

The key differentiators for vercel-labs/portless are: (1) agent-first design with SKILL.md integration, (2) zero-config for frameworks respecting `PORT` env var, (3) active Vercel-backed development, and (4) HTTP/2 support for modern dev servers.

The closest predecessor is typicode/hotel, which peaked at ~10K stars and offered a similar `.localhost` domain concept with a web UI process manager. Hotel appears unmaintained and used a PAC (Proxy Auto-Configuration) approach rather than a direct reverse proxy, which was more fragile and required browser configuration.

## Recommendations

**For individual developers:** Install and try it now. The tool is simple enough that the risk is minimal -- it's just a local proxy with no cloud dependency. If you work with monorepos or multiple local services, the naming convention alone (`api.myapp.localhost:1355` vs `localhost:3001`) improves developer experience significantly.

**For AI-assisted development:** This is where portless shines most. If you use Claude Code, Cursor, Codex, or other AI coding agents in monorepo environments, portless eliminates the class of errors where agents guess wrong ports. The SKILL.md integration means agents can learn about portless automatically.

**For teams:** Wait for v1.0 and broader community validation. The single-contributor risk is real, though Vercel Labs backing mitigates it somewhat. Windows support (#6) and config file support (#13) should land before team-wide adoption.

**Watch for:** Whether Vercel integrates portless into their platform or `vercel dev` command. The Vercel Labs designation suggests experimental status, but the agent-first angle aligns with Vercel's broader AI strategy (skills.sh, AI SDK, agent-browser).
